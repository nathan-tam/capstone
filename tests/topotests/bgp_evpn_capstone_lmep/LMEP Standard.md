Standard: Layer-2 Mapping & Encapsulation Protocol (LMEP)<br>
Draft Version: 1.0<br>
Status: Functional Specification<br>
Focus: MAC-to-VTEP Direct Translation & Mobility Management<br>

### 1. Protocol Objective
The LMEP standard defines a method for replacing traditional MAC learning (flood-and-learn) with a deterministic unidirectional Mapping Service that intercepts controller traffic and encapsulates it into VXLAN packets, based on a centralized MAC-to-Locator database. 
This mapping server will be used as the intermediary forwarding device in the overlay network. The server will not consist of pull or EVPN push logic but rather be preemptively updated by the sign-on of clients to the WAPs. The database that will hold all the endpoint locations of the robots will not be shared with other VTEPs. The controller data packets will flow through the server and be routed to the correct pseudo tunnel.

### 2. Architectural Components
The Listening Port: A controller-facing logical interface on the Mapping Server responsible for intercepting Ethernet frames, performing MAC-to-VTEP resolution, and encapsulating eligible traffic into VXLAN for transport across the overlay.
Mapping Server (MS): The authoritative database storing the {MAC:VTEP_IP} relationship. Multiple Mapping servers can live in an HA or load-balancing configuration by using multicast to sync the reception of MAC registration messages.
WAP: The wireless access point that sends a MAC registration packet upon a client beginning to register.
VTEP: The VxLAN terminating-point to the overlay network, used for encapsulating at the listening port and de-encapsulating at the WAPs connected to the robots.

### 3. The Control Plane: MAC Registration 
When the device begins registration with the WAP, it will generate a MAC registration packet. That will be sent to the mapping server, which will register the new location of the device.
The WAP must immediately send a MAC registration message to reduce traffic "black-holing." Since VxLAN and IP in general are considered best effort, there will still be lost packets, but it is significantly reduced since the pseudo-tunnel should come up before the host has finished registering with the WAP. 

##### 3.1 Mac-Register Message Structure (TLV)
Below is the custom MAC registration packet TLV that will be generated by the WAP that the robot is connected to:

| Type | Length | Value | Description |
|---|---|---|---|
| 0x01 | 6 Bytes | Client MAC | The Mac is being registered to a new VTEP |
| 0x02 | 4 Bytes | Client IP | The IP of the client [cite: 24] |
| 0x03 | 3 Bytes | VNI | The VXLAN Network Identifier. |
| 0x04 | 4 Bytes | Vtep IP | The IP of the VTEP that controls the L2 domain in which the WAP resides in |

### 4. The Data Plane: Intercept & Translate
The "Listening Port" on the mapping server operates using the following logic for every incoming packet:
##### 4.1 Interception (Controller to Robot)
1. Controller Instruction: The controller emits plain Layer-2 Ethernet frames addressed directly to the robot’s MAC address.
2. Intercept: The port facing the controller captures the frame.
3. Lookup: The mapping server queries the L2-Map.
4. Encapsulation:
    * If a destination vtep is found for the MAC, the frame is wrapped in a VXLAN Header.
    * The Inner Destination MAC remains the client's MAC.
    * The Outer Destination IP is set to the remote VTEP.
5. Forward: The packet is sent through the VXLAN fabric to the destination VTEP, where it is de-encapsulated and forwarded to the host. 


##### 4.2 Returning (Robot to Controller)
Packets will be returned through asymmetric routing, as they don't need to go through the LMEP system; as the controller is not mobile, they can go through normal VXLAN processing or even IP routing.

### 5. Mobility & Roaming Logic
To handle high-speed roaming, LMEP utilizes Pre-emptive Updates:
* Step 1: Client starts to register with a WAP
* Step 2: WAP sends a MAC registration message to the mapping server indicating its new location
* Step 3: Mapping server receives the MAC registration message and updates its L2-map
* Step 4: Mapping proceeds as normal with the updated MAC

### 6. Clean Up Process
##### 6.1 Default process
Mapping entries will be removed when they age out from either not being used to translate packets in a set amount of time, and if they have not moved in a while. The specific values will be user-tunable, with defaults to be determined.
##### 6.2 After an entry ages out
After an entry has reached its age-out time and is removed one of the following actions can be taken to ensure the robot is not operational to mitigate soft blocks (Robot is active but does not receive instructions, making it nonoperational):
* Mapping entries will be marked as stale with a user-configurable idle timer, and will be removed if we send a ping or some other form of keepalive and there is no reply.
* Mapping entries will be marked as stale with a user-configurable idle timer, and we will reach out to the WAP for a list of its active clients and see if the MAC address is inside of it.

# HIGHERARCHICAL LMEP
### 1. Conceptual Framework
H-LMEP is designed for environments with a fixed hierarchy (Root-to-Leaf) where "Uplink" and "Downlink" ports are manually defined to prevent loops. It treats the network as a directed acyclic graph (DAG). Instead of maintaining global routing tables, the network state is updated reactively by the mobile robots as they change their attachment points.
### 2.The Packet Header
The protocol utilizes a simplified shim header to minimize overhead and maximize switching speed, similar to an MPLS label stack but optimized for one-way flows.
| Field | Size | Description |
|---|---|---|
| Type | 4 bits | Identifies the packet as DATA, MACREG, or TEARDOWN. |
| Label/Path ID | 20 bits | The locally significant identifier used for high-speed forwarding. |
| Target MAC | 48 bits | The unique identifier of the destination mobile robot. |
| Timestamp | 32 bits | Timestamp, unix time (can be made smaller cause we def dont need that much precision) |

### 3. Operational Logic
The core innovation is the behavior of the `macreg` packet as it travels toward the root.
##### Step 1: Path Registration
1. **Origin**: A mobile robot connects to a router and sends a macreg.
2. **Ingress**: The router receives the macreg on a Downlink port.
3. **Table Lookup**: The router checks if the Target MAC exists.
    * If found: It updates the entry to point to the new Downlink port.
    * If not found: It creates a new entry mapping the Target MAC to that Downlink port.
4. **The Swap**: The router replaces the packet's destination info with the path data required for the next hop.
5. **Uplink Forwarding**: The packet is sent out the Uplink port that has the lowest cost to get to the controller. A router forwards a `macreg` uplink only if it creates a new entry for the Target MAC. If the router already has an entry and must modify it to point to a different downlink port, it updates locally and ends the propagation.

##### Step 2: Data Forwarding
Once the path is "carved," the Root Node can send data packets. The routers simply look at the label, see which Downlink port it maps to, and switch the packet. No complex lookups are required at high speeds.

### 4. Mobility and Table Eviction
The eviction policy focuses on Path Correction.
* Reactive Teardown: When a macreg arrives at a router that already has an entry for that MAC—but on a different Downlink port—it realizes the robot has moved.
* Correction: The router updates its own table for the new port. Simultaneously, it sends a TEARDOWN message down the old Downlink port to prune the stale path and free up resources in lower-level switches.
* Broken path resolution: if a router gets a teardown message with a timestamp thats older than its one, it ignores it and resends its path message
### 5. Summary of Benefits
* Speed: MPLS-like switching on the downlink.
* Simplicity: No Spanning Tree Protocol (STP) or complex OSPF/BGP overhead.
* Efficiency: CAM tables only store paths for robots that are actually active and connected.
